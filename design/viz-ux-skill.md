# 🔬 可视化互动模块设计 Skill

> 从打磨得最到位的三个模块（**池化 / 边缘检测滑窗 / K近邻投票**）反推出的配方。
> 用途：把任意一个抽象 AI 概念，变成一个"亲手玩几下就懂"的可视化互动模块。最终落地内容远不止 20 个，靠这套配方**举一反三批量产出**。

---

## 一、什么叫"到位"的可视化模块（判断标准）

拿三个标杆对照，它们共同做对了：

| 模块 | 抽象概念 | 变成了什么可玩的动作 | 为什么到位 |
|---|---|---|---|
| 池化 | 降采样、保留特征 | 点一下，每个 2×2 块只留最大值，逐层缩小 | **看得见"缩"的过程**，一步步 |
| 边缘检测滑窗 | 卷积找特征 | 拖小窗口扫过图，扫到边缘就变绿变亮 | **亲手扫**，因果即时（到边缘→亮） |
| K近邻 | 相似度分类 | 点空白放个新点，最近 3 个邻居高亮投票 | **自己放点**，结果当场揭晓 |

提炼出的**「到位」= 5 个必备特征**：
1. **一个抽象概念 → 一个能亲手操作的具体隐喻**（不是看，是玩）。
2. **操作驱动，非阅读驱动**：核心是"拖/点/滑一下 → 立刻看到变化"，文字只是标注。
3. **过程可见**：把抽象运算拆成**一步步 / 一层层 / 一格格**的可见动作。
4. **因果即时且夸张**：动作→结果的反馈要马上、直观、放大（高亮、变色、表情、数字跳动）。
5. **不看文字也能大致玩懂**：一句指令 + 明显的可点线索（👆/黄框/闪烁）。

## 二、设计流程（6 步，每造一个模块走一遍）

1. **抽概念的"一句话内核"**：这个概念最核心的一个动作是什么？（池化=留最强；KNN=看邻居；梯度下降=往低处走）
2. **找可操作隐喻**：把内核映射成一个孩子能"手动执行"的动作（滑窗、放点、走一步、拖滑块）。
3. **定操作方式**：拖 / 点 / 滑 / 长按 —— 选**最能体现内核**的那一种。
4. **设计"过程可见"**：怎么把运算拆成看得见的步骤/层次/高亮。
5. **加即时反馈**：变色 / 高亮 / 表情 / 数字 / 撒花，让每次操作都有"响"。
6. **诚实边界 + 参数化**：一句话点破隐喻哪里不完全对；数据/题目外部传入，一套交互复用到多个知识点。

## 三、交互原型分型（选型清单）

按"概念的内核动作"选原型：

| 内核动作 | 推荐原型 | 例子概念 |
|---|---|---|
| 分类 / 归堆 | 拖拽分拣、K近邻放点、聚类 | 分类、相似度 |
| 扫描 / 找特征 | 滑窗、放大镜 | 卷积、像素 |
| 逐层压缩 / 变换 | 逐层网格链 | 池化、降维 |
| 逐步逼近 / 优化 | 走一步（下山）、逐步去噪 | 梯度下降、扩散生成 |
| 阈值 / 开关 | 滑块过线、神经元激活 | 感知机、激活函数 |
| 传播 / 流动 | 脉冲逐层 + **必须画节点间连线**（连线本身＝概念） | 前向传播、注意力、反向传播 |
| 序列 / 拆分 | 切块、接龙、采样 | 分词、猜下一个词、温度 |
| 权衡 / 调参 | **单变量实时探索**：只拖一个滑块 → 立刻重绘整条轨迹/曲线（小/刚好/过大→发散） | 学习率、过拟合、温度 |

## 四、技术实现要点（踩过的坑）

- **防网页拉伸错位**：位置判定一律用**百分比坐标 / 容器相对坐标**（`offsetX/clientWidth`），不用绝对像素+角度（黄金矿工的教训）。
- **动画**：连续动效用 `requestAnimationFrame`；纯 DOM + `transform/opacity` 优先，几何/曲线才上 `<canvas>`。
- **SVG 连线**：用 `viewBox="0 0 100 100" preserveAspectRatio="none"`，节点用 % 定位，坐标天然对齐。
- **统一外壳**：套 `.demo-stage` / `.ixbox` / `.gameframe`；反馈用 `bigFlash()`、`spark()`。
- **每个模块自带重置**：框架已统一注入 ↻。
- **演示页也默认浅色工作台**：黑底只用于首页星空/庆祝/极少数短时"黑箱"效果；课程内、UX demo、可视化原型默认用浅色 `.ixbox`/`.demo-stage`，避免儿童长时间阅读深色界面。
- **禁用暗色模式漂移**：页面级加 `color-scheme: light`；按钮、标签、提示文字必须显式声明前景色和背景色，不能依赖浏览器默认按钮样式。
- **可读性优先于氛围**：浅色舞台里避免大块深色容器；若某个图形必须保留深色对比，只保留在局部画布/小图元内，周围说明、按钮、计数器仍走浅色系统。

## 七、交互打磨 Checklist（从「学」章活切片迭代提炼 · 交付前逐条过）

这些是真实迭代里踩出来的，做每个交互/章节都对照检查：

**A. 交互完整性**
- ☐ 每个交互都有 **↻ 重置**（点了能重来）。
- ☐ 选择类交互能 **取消/反选**（再点一次取消选中）。
- ☐ 多步交互按**因果顺序串起来**（如"骗过 AI"= 先给看✓→偷改→再给看✗），**不要把不同层级的动作平铺成同级按钮**。
- ☐ 完成有明确终点，不会卡死。

**B. 显性奖励（与金币货币打通）**
- ☐ 每完成一个交互，**金币从点击处蹦出→飞向右上角计数器**，计数器跳动。
- ☐ 通关给大奖 + 徽章掉落 + 解锁下一站。
- ☐ 失败**暖色鼓励、零惩罚**，绝不红叉。

**C. 难度手感**
- ☐ 有"爬坡感"：如训练机增量**线性递减、封顶 100%、越高越难、点击量递增**，别一两下就满。

**D. 数据可视化别"定死"**
- ☐ 数据点**足够多、够散、够清晰**（加描边/加亮），避免"看着就三个点/没变化"。
- ☐ 让要教的对比**一眼可见**（如欠拟合直线明显漏点 vs 过拟合扭线穿每点）。

**E. 文本 & 术语**
- ☐ 正文用 `parseP` 走 **innerHTML**，支持 `<b>` 等，**绝不把标签当文字露出 `<>`**。
- ☐ 术语双段**只给晦涩词**，第二段给真深度（成因/机制/意义），**不重复第一段**；简单词只留大白话。
- ☐ 中文常用词不加英文；Transformer/CNN/RNN/LLM/Agent 等常用英文词给中英对照。无括号、无前缀。

**F. 技术稳健**
- ☐ 位置判定用**百分比/容器相对坐标**，防网页拉伸错位。
- ☐ `setInterval`/`requestAnimationFrame` **用完清理**（通关、离开、重置都 clear），防泄漏。
- ☐ 静态页无法逐帧渲染验证 → **JS 语法校验（new Function）+ 每个交互 try/catch 兜底 + 人工预览走查**。
- ☐ 用 Chromium/真实浏览器截图检查浅色/深色可读性，尤其是按钮文字、头部标题、内联 `style` 写死的颜色。
- ☐ 若截图里出现黑底黑字、浅底白字、按钮文字消失，优先改组件级 CSS，不要用过宽的 `*` / `div:not(...)` 兜底选择器。

## 八、可视化语义铁律（进阶课并入主页·第二轮打磨提炼 · 2026-07）

这一轮全是"能动但不对"的问题——交互能跑，却在语义/结构/适配上翻车。交付可视化前额外过这 5 条：

**1. 隐喻方向必须和概念一致（最容易翻车）**
- 梯度**下降**就要真的往**下、往低处**走：把 loss 画成**山谷（U 形，谷底在下方）**，小球滚进谷底。曾把山谷画反成"山峰"，小球往上爬 → 直接和"下降"矛盾。
- 通用检查：概念里的"增/减、上/下、聚/散、快/慢"，在画面上的方向要**逐一对得上**，别让视觉和文字打架。

**2. 结构性视觉元素不能省 —— 连线就是概念本身**
- 讲"传播/流动/注意力"，**节点之间的连线必须画出来并随信号点亮**；只画一排排孤立圆点＝把"网络"画成了"散点"，概念丢失。
- 一般化：凡是"关系/连接/流向"类概念，承载关系的那根线/箭头是主角，不是装饰。

**3. 图表默认响应式：用 SVG `viewBox`，不用定宽像素**
- 折线/曲线/网络图一律 `<svg viewBox="0 0 W H" width="100%">`，坐标在 viewBox 空间里算 → **任意屏宽自动缩放**。
- 定宽 flex 柱状（`width:12px` 一根根）在窄屏溢出、宽屏稀疏；能画成 SVG 折线就别用定宽柱。

**4. "调参"类优先做成单变量实时探索**
- 让玩家**只调一个变量**（如学习率），拖动即**实时重绘整条结果轨迹**，一眼看全"太小→慢 / 刚好→收敛✅ / 太大→横跳 / 过大→发散"。
- 通常比"点一次走一步"信息量更大、更有掌控感；"走一步"适合强调"逐步过程"，"实时探索"适合强调"参数的影响"。

**5. 少堆装饰性小标签；章末别复用同一个街机**
- 每节别再挂一个"左下角心智桩"小胶囊——视觉碎片。**关键锚点折进正文收束句或课程小结卡**即可。
- 章末"通关挑战"**别把同一个落物街机换皮复用**（跨章重复＝无聊）。要么按本课内容做**专属解谜**（借鉴独立/解谜游戏本质：调参达标、连线通路、约束满足、最短路径…），要么用**"课程小结·领徽章"收尾卡**代替，宁缺毋滥。
- 首屏留白：内容首卡与顶部标题栏之间留出间距，别贴住/重叠。

> 一句话总纲：**可视化 = 语义正确（方向对）＋ 结构完整（连线在）＋ 宽度自适应（viewBox）＋ 操作有效（单变量实时）**。四者缺一，就是"能动但不对"。

## 九、信息节奏：逐段展开 & 交互落位（2026-07 用户测试后新增·重要）

用户测试暴露两个真实问题，都关乎**信息呈现的节奏**（不是单个组件好不好，而是"一节里各块怎么排、何时出现"）：

**1. 交互别放最前面 —— 按"阅读理解链"落位**
- 一节的顺序应是：**标题/对话（抛出知识点）→ 解释正文（讲机制）→ 交互（直观理解）→ 收束阐述（命名术语）**。
- 交互是"读懂之后的印证与内化"，不是开胃菜。放最前面 → 孩子直接玩、跳过正文。**交互固定落在解释正文之后、收束之前。**

**2. 逐段展开（视觉小说式）—— 一次只给一个信息单元**
- 问题：文字与图像/交互**同时出现**，孩子注意力被游戏吸走，正文没读进去。
- 方案：一节内容**分段揭示**——先出"标题 + 对话"，其余每个段落 / 交互用**小灰色下箭头 `▽` 引导点击**逐个展开（模仿视觉小说 ADV 一句一点）。同一时刻"新出现"的信息只有一块，注意力不被撕裂。
- 与术语心灵旅程对齐：每个 beat = 弧线一步（遇见→干嘛→运转→交互→掌握）。
- **复习豁免**：这一节**学过后再进入（复习）就全部直接展开**，不再逐段点击（用 `ST.collected[ch.n]` / `ST.done[ch]` 判定）。
- 箭头样式：低饱和灰、居中、轻微上下浮动提示可点；点击后当前箭头消失、下一块 `slideUp` 淡入、箭头下移；最后一块出现后换成"下一节/完成"按钮。

> 一句话：**硬科普要让孩子读进去，就得控制信息节奏——按理解链排序、逐段揭示、复习全展开。**
